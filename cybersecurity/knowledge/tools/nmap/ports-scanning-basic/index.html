    # Port states

    Nmap considers the following states for ports:

    1. Open - a service is listening on the specified port
    2. Closed - no service is listening on the specified port, although the port is accessible (it is reachable and is not blocked by a firewall or other security appliances/programs)
    3. Filtered - Nmap cannot determine if the port is open or closed because the port is not accessible.
      • usually due to a firewall preventing Nmap from reaching that port
      • either Nmapâ€™s packets are blocked from reaching the port or the responses are blocked from reaching Nmapâ€™s host
    4. Unfiltered - Nmap cannot determine if the port is open or closed, although the port is accessible.
      • encountered when using an ACK scanÂ -sA
    5. Open|Filtered - Nmap cannot determine whether the port is open or filtered
    6. Closed|Filtered - Nmap cannot decide whether a port is closed or filtered

    # TCP Connect scan

    TCPÂ connect scan works by completing the TCP 3-way handshake (SYN -> SYN/ACK -> ACK). This might make the logs to appear on the target device.

    We are interested in learning whether theÂ TCPÂ port is open, not establishing a TCP connection. Hence the connection is torn as soon as its state is confirmed by sending a RST/ACK. We can choose to run TCP connect scan usingÂ nmap -sT ....

    As a non-privileged user, aÂ TCPÂ connect scan is the only possible option to discover open TCP ports.

    Nmap sendsÂ TCPÂ packets with SYN flag set to various ports, 256, 443, 143, and so on (1000 most common ports).     # TCP SYN scan

    The default scan mode is SYN scan, and it requires a privileged user to run it and is a very reliable option.

    SYN scan does not need to complete the TCP 3-way handshake; instead, it tears down the connection once it receives a response from the server. Because we didnâ€™t establish a TCP connection, this decreases the chances of the scan being logged (SYN -> SYN/ACK -> RST)

    We can select this scan type by using theÂ -sSÂ option.

    # UDP Scan

    UDPÂ is a connectionless protocol, and hence it does not require any handshake for connection establishment.

    We cannot guarantee that a service listening on a UDP port would respond to our packets. Therefore, sending a UDP packet to an open port wonâ€™t tell us anything.

    However, if a UDP packet is sent to a closed port, an ICMP port unreachable error (type 3, code 3) is returned. The UDP ports that donâ€™t generate any response are the ones that Nmap will state as open.

    # Ports to scan
